# ID-04.2 — Assign items to categories

Issue: ID 4.2

Files to know:
- Backend API: `listme-backend/src/main/java/com/oliwier/listmebackend/api/ItemController.java`
- Backend DTOs: `.../api/dto/CreateItemRequest.java`, `.../api/dto/UpdateItemRequest.java`, `.../api/dto/ItemResponse.java`
- Domain: `items.category_id` FK (see V3 migration)
- Frontend service: `listme-frontend/src/services/item.ts`
- Frontend store: `listme-frontend/src/stores/items.ts`
- Frontend UI: `components/item/AddItemSheet.vue` (picker), `components/item/ItemRow.vue` (chip)

---

## Overview

Each item can be linked to a category. This is optional. When an item has a category, the UI shows a small chip with the category name (and color, if provided) under the item name.

---

## Data model

- Item fields: `categoryId: string | null`, `categoryName: string | null`, `categoryColor: string | null`.
- Changing a category affects only these fields and does not change the item `position` or `checked` state.

---

## Backend API

- Create item with a category:
  - POST `/api/lists/{listId}/items`
  - Body: `{ name: string, categoryId?: string, quantity?: number, quantityUnit?: string, price?: number, imageUrl?: string, labelIds?: string[] }`
- Update item category (and/or other fields):
  - PUT `/api/lists/{listId}/items/{itemId}`
  - Body: `{ name: string, categoryId?: string, ... }`
- Responses include resolved `categoryName`/`categoryColor` when applicable (see `ItemResponse`).

Authorization: The `@CurrentDevice` must be a participant of the list. Otherwise 403.

---

## Frontend flow

- Add Item: The `AddItemSheet` offers a category picker. When submitted, the `itemsStore.create(listId, req)` is called with `req.categoryId` set.
- Edit Item: The edit form allows changing the category and saves via `itemsStore.update(listId, itemId, req)`.
- Display: `ItemRow.vue` renders a chip when `item.categoryName` exists. If `categoryColor` exists, the chip is tinted to the color.

---

## Caching & offline

- On success, Pinia store writes through to IndexedDB via `CacheService.saveItem(item)`.
- Offline create uses a temporary UUID and persists it locally with the selected `categoryId` (see `items.ts` offline path). The actual category association is synchronized when back online via the OperationQueue.
- If a category is deleted later, affected items keep their integrity; their `categoryId` becomes `null` and chips disappear (due to `ON DELETE SET NULL`).

---

## Edge cases

- Category deleted: Items assigned to a removed category are still valid (chip disappears).
- Unknown categoryId on update: API will reject or normalize; ensure the picker uses valid IDs from `/api/lists/{listId}/categories`.
- Moving items between categories does not change sort order unless the UI sorts by category (see 4.4 — not implemented yet).

---

## Manual verification

1) Create an item assigned to a category
- Precondition: one category exists (e.g., "Dairy").
- Action: POST create with `categoryId` set.
- Expect: Response has `categoryId` and `categoryName`; UI shows chip “Dairy”.

2) Update an item to a different category
- Action: PUT update with a different `categoryId`.
- Expect: UI chip updates to the new category immediately; persists across reload (IndexedDB).

3) Delete the category
- Action: DELETE the category.
- Expect: Items remain, chips disappear; no errors when loading the list.
