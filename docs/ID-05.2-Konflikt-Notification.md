## Overview

When concurrent operations are detected during a live sync session, the user is notified through two layers:

| Layer | Component | Scope |
|-------|-----------|-------|
| **In-list banner** | `ConflictBanner.vue` | Visible only while the affected list is open (see [ID-05.3](ID-05.3-Konflikt-Resolution.md)) |
| **Global toast** | `ConflictToast.vue` | Visible app-wide, including while navigating away — persisted in Pinia |

This document covers the global toast notification (ID-05.2). The in-list banner is documented in [ID-05.3](ID-05.3-Konflikt-Resolution.md).

---

## Notification Store

**File:** `stores/notifications.ts`

A Pinia store that acts as a queue for global conflict notifications. Any part of the app can add a notification; `ConflictToast.vue` reads and renders them.

```ts
interface ConflictNotification {
  id: string       // auto-generated UUID
  listId: string
  listName: string
  count: number    // number of new conflicts in this batch
  ts: number       // Date.now() at time of creation
}
```

### Store actions

| Action | Description |
|--------|-------------|
| `add(n)` | Pushes a new notification, generating `id` and `ts` automatically |
| `dismiss(id)` | Removes a single notification by ID |
| `dismissAll()` | Clears all notifications (called on mount to remove stale entries) |

## ConflictToast

**File:** `components/common/ConflictToast.vue`

A `<Teleport to="body">` component that renders a floating notification stack above the bottom navigation bar (`bottom-24`). It reads directly from the `notificationsStore`.

### Visual design

Each toast card sits in a stack at the bottom of the screen:
- Yellow `⚠` icon on the left
- **"Konflikt automatisch gelöst"** as the heading
- List name truncated to one line below
- `✕` dismiss button on the right

The component uses `<TransitionGroup>` so cards animate in from below and fade out when dismissed.

### Auto-dismiss

A `watch` on `store.notifications.length` starts a `setTimeout` for the oldest notification whenever the array grows. The delay is `6000ms − age` — meaning if a notification was pushed 2 seconds ago, the timer waits only 4 more seconds. This prevents rapid batches from all expiring simultaneously.

```
notification pushed → watch fires → oldest age checked
  age < 6s → setTimeout(6s - age)
  age ≥ 6s → setTimeout(0) — dismiss immediately
```

### Lifecycle

`onMounted` calls `store.dismissAll()` to clear any notifications left over from a previous navigation (e.g. if the user closed the app mid-session). This prevents stale toasts from appearing when reopening the app.

## How a notification is created

Notifications are triggered inside `useListSync` whenever `detectConflicts()` finds more conflicts than were present before the current operation:

```ts
const newConflicts = detectConflicts(sessionOps)
const hadConflicts = conflicts.value.length

conflicts.value = newConflicts

if (newConflicts.length > hadConflicts) {
  notificationsStore.add({
    listId,
    listName: '',   // list name not available in this scope
    count: newConflicts.length - hadConflicts,
  })
}
```

The `listName` field is intentionally left empty in the composable because the composable does not have access to the list metadata. The toast handles an empty `listName` gracefully by showing only the heading line.

## End-to-end flow

```
Device B pushes a concurrent op
        │
        ▼
useListSync: WebSocket callback fires
        │
        ▼
detectConflicts(sessionOps) → new conflicts found
        │
        ├─── conflicts.value updated  ──→  ConflictBanner (in-list)
        │
        └─── notificationsStore.add() ──→  ConflictToast (global)
                                                │
                                        auto-dismiss after 6s
                                        OR user taps ✕
```
