# ID-01.2 — CacheService

**Issue:** #24
**File:** `listme-frontend/src/services/cache.ts`

---

## Overview

`CacheService` is a write-through cache that keeps a local copy of lists and items in IndexedDB (via Dexie). It is the bridge between the Pinia stores and the `listme-cache` IndexedDB database.

**Write-through** means: every time the app successfully receives data from the API, it immediately writes a copy to IndexedDB. On next load (or on a failed API call), the Pinia stores read from IndexedDB instead of blocking on the network.

---

## Why it exists

Without `CacheService`, the app would show a blank screen or spinner whenever:
- The device is offline
- The API takes a long time to respond
- The user reloads the page before the API responds

With `CacheService`, lists and items are available from IndexedDB within milliseconds, before any network request completes.

---

## Cache strategy per store

### lists store (`src/stores/lists.ts`)

```
fetchAll() called
    │
    ├── API call fires (async)
    │
    ├── CacheService.getLists() fires immediately
    │       │
    │       └── sets store from cache → UI renders instantly
    │
    └── API resolves
            │
            └── CacheService.saveLists() → updates cache
                sets store from API response → UI re-renders with fresh data
```

### items store (`src/stores/items.ts`)

Same pattern. `fetchAll(listId)` populates from cache first, then overwrites with the API response and saves it back.

## API reference

### Lists

#### `saveLists(lists: ShoppingList[]): Promise<void>`

Bulk-writes an array of lists to IndexedDB. Adds `_savedAt = Date.now()` to each row. Uses `bulkPut` so existing rows are overwritten rather than duplicated.

```typescript
await CacheService.saveLists(listsFromApi)
```

#### `saveList(list: ShoppingList): Promise<void>`

Writes a single list. Used after create or update operations.

```typescript
await CacheService.saveList(newList)
```

#### `getLists(): Promise<ShoppingList[]>`

Returns all cached lists ordered by `_savedAt` descending (most recently synced first). Strips the internal `_savedAt` field before returning.

```typescript
const cached = await CacheService.getLists()
```

#### `removeList(listId: string): Promise<void>`

Deletes a list and **all its items** from IndexedDB. Called when the user leaves or deletes a list.

```typescript
await CacheService.removeList(listId)
```

---

### Items

#### `saveItems(listId: string, items: Item[]): Promise<void>`

Replaces the entire item snapshot for a list. First deletes all existing items for that `listId`, then bulk-inserts the new array. This keeps the cache in sync when the server reorders items or returns deletions.

```typescript
await CacheService.saveItems(listId, itemsFromApi)
```

#### `saveItem(item: Item): Promise<void>`

Writes a single item. Used after a create, update, or check/uncheck operation.

```typescript
await CacheService.saveItem(updatedItem)
```

#### `getItems(listId: string): Promise<Item[]>`

Returns all cached items for a list, sorted by `position` ascending. Strips `_savedAt`.

```typescript
const cached = await CacheService.getItems(listId)
```

#### `removeItem(itemId: string): Promise<void>`

Deletes a single item from the cache. Called on swipe-to-delete.

```typescript
await CacheService.removeItem(itemId)
```

#### `patchItem(itemId: string, changes: Partial<Item>): Promise<void>`

Updates specific fields of a cached item without rewriting the whole object. Used for optimistic updates — e.g. toggling `checked` updates only that field.

```typescript
await CacheService.patchItem(itemId, { checked: true })
```

---

## Data flow diagram

```
User action (e.g. toggle checkbox)
        │
        ▼
itemsStore.toggleCheck(itemId)
        │
        ├── optimistic: update Pinia store immediately
        │
        ├── CacheService.patchItem(itemId, { checked })  ← survives reload
        │
        └── API call: PATCH /api/lists/{id}/items/{itemId}/check
                │
                └── success: CacheService.saveItem(serverItem)  ← sync server timestamp
```

---

## What is NOT cached

- **CRDT operations** — stored in the separate `listme-opqueue` database (see [ID-01.3-SyncQueue.md](ID-01.3-SyncQueue.md))
- **Labels** — currently fetched fresh on each open (small payload, rarely changes)
- **Presets** — not cached; only used when creating a list (one-time operation)

---

## Clearing the cache

To wipe the cache during development:

```javascript
// In browser DevTools console:
const db = new Dexie('listme-cache')
await db.delete()
location.reload()
```

Or via DevTools → Application → IndexedDB → `listme-cache` → right-click → Delete database.
