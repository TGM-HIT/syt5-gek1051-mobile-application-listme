# ID-04.5 — Create item

Issue: ID 4.5

Files to know:
- Backend API: `listme-backend/src/main/java/com/oliwier/listmebackend/api/ItemController.java`
- Backend DTOs: `.../api/dto/CreateItemRequest.java`, `.../api/dto/ItemResponse.java`
- Frontend service: `listme-frontend/src/services/item.ts`
- Frontend store: `listme-frontend/src/stores/items.ts`
- Frontend UI: `listme-frontend/src/components/item/AddItemSheet.vue`
- Caching: `listme-frontend/src/services/cache.ts` (CacheService)

---

## Overview

Users can add new items to a shopping list. Items may include an optional category, quantity/unit, price, image, and labels. The app is offline-first: created items appear immediately and are persisted to IndexedDB; if offline, the create is queued for synchronization.

---

## Backend

- Endpoint: `POST /api/lists/{listId}/items`
- Request body (see `CreateItemRequest`):
  Example JSON (comments removed):
  ```json
  {
    "name": "Milk",
    "categoryId": "00000000-0000-0000-0000-000000000000",
    "labelIds": ["11111111-1111-1111-1111-111111111111"],
    "quantity": 2,
    "quantityUnit": "L",
    "price": 1.29,
    "imageUrl": "https://example.com/milk.png"
  }
  ```
- Response: `ItemResponse` with server-assigned `id`, resolved `categoryName`/`categoryColor` when applicable, and timestamps.
- Validation: `name` is required and limited to 500 characters.

---

## Frontend flow

- UI: `AddItemSheet.vue` collects the fields and calls the store action.
- Store action: `itemsStore.create(listId, req)` →
  - On success (online):
    - Pushes the returned item into the list
    - `CacheService.saveItem(item)` persists it locally
    - Updates list counters (checked/total)
  - On network error (offline):
    - Creates a client-side item with a temporary UUID (`crypto.randomUUID()`)
    - Persists it via `CacheService.saveItem(item)`
    - Enqueues an `ITEM_CREATE` operation in `OperationQueue` for later sync

---

## Caching & offline details

- Items are stored in `listme-cache` IndexedDB. On next launch (even offline), they are rendered immediately (`CacheService.getItems(listId)`).
- The service worker does not cache API responses; dynamic data uses IndexedDB (see ID-01.4).
- The vector clock and CRDT operation queue ensure correct conflict resolution during sync.

---

## Edge cases

- Duplicate names are allowed by design; users often add similar products.
- Very long names (>500 chars) will be rejected by the backend; the UI should trim or show an error.
- If a stale category is chosen (deleted in the meantime), backend will normalize; UI should refresh categories when the AddItemSheet opens.

---

## Manual verification

1) Create online
- Open a list → add "Milk" with category "Dairy" → save
- Expect: Item appears instantly with a “Dairy” chip; after reload, still present

2) Create offline
- Disable network → add "Apples" → item appears instantly
- Reload offline → item persists (from IndexedDB)
- Re-enable network → item is synced (no duplicates)

3) Validation
- Try to submit an empty name → expect client-side prevention or server 400
