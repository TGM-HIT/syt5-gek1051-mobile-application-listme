## Overview

Within the share modal, users can reveal a scannable QR code for a list's share link. This is useful for handing a phone to someone nearby instead of copying and pasting a URL. The QR code is generated client-side from the share URL using the `qrcode` library and rendered to a `<canvas>` element.

---

## Frontend

### ShareListModal.vue

The QR code is one of two share options inside the existing share modal (the other being "Link kopieren").

**Props:**

| Prop          | Type           | Description                          |
| ------------- | -------------- | ------------------------------------ |
| `modelValue`  | `boolean`      | Controls modal open/closed state     |
| `list`        | `ShoppingList` | The list being shared                |

**Emits:**

| Event                | Payload          | Description                              |
| -------------------- | ---------------- | ---------------------------------------- |
| `update:modelValue`  | `false`          | Emitted when the modal is closed         |
| `token-changed`      | `string \| null` | Emitted after token is generated/revoked |

**QR code flow:**

1. Modal opens â†’ if `list.shareToken` is null, `shareService.generateToken(listId)` is called and the returned token is stored.
2. The share URL is constructed: `${window.location.origin}/s/${token}`.
3. User taps **QR-Code** button â†’ `showQr` toggles to `true`.
4. `nextTick` â†’ `QRCode.toCanvas(canvasRef, shareUrl, options)` is called:
   - `width: 200`
   - `margin: 2`
   - `color.dark: '#303446'` (Catppuccin Frappe base)
   - `color.light: '#eff1f5'` (Catppuccin Latte base)
5. The canvas is shown inside the modal; tapping **QR-Code** again hides it.

**Token lifecycle:**

- Token is generated once on modal open (if none exists) and reused on subsequent opens.
- **Widerrufen** button calls `shareService.revokeToken(listId)`, clears the local token, and emits `token-changed(null)`.
- Revoking makes the old share link and QR code invalid immediately.

### shareService (`src/services/share.ts`)

| Method                   | HTTP call                      | Purpose                        |
| ------------------------ | ------------------------------ | ------------------------------ |
| `generateToken(listId)`  | `POST /lists/:id/share`        | Create/return share token      |
| `revokeToken(listId)`    | `DELETE /lists/:id/share`      | Invalidate existing token      |

---

## No backend changes for QR

QR code generation is entirely client-side. The backend only stores and validates the share token (unchanged from US06). The QR code is just a visual encoding of the share URL â€” scanning it is equivalent to opening the link.

---

## Edge cases

- **No token yet:** `generateToken` is awaited before the QR section can be shown; the QR button is visible but the canvas only renders after the token resolves.
- **Token revoked while QR visible:** After revoke the `shareUrl` becomes empty and `showQr` resets to `false`; the canvas is unmounted.
- **QR library unavailable / canvas error:** The `toCanvas` call is not wrapped in try/catch in the current implementation â€” a failed render would leave the canvas blank. The user can still use the copy-link fallback.
- **Dark mode:** QR colours are hardcoded to the Catppuccin palette and do not change with the theme toggle.

## Manual verification

1. **Happy path**
   - Open a list â†’ tap the share icon â†’ tap **QR-Code** â†’ expect a QR code canvas to appear
   - Scan with another device â†’ expect the share link to open and the list to be added

2. **Toggle hide**
   - Tap **QR-Code** again â†’ expect the canvas to disappear

3. **Revoke then QR**
   - Tap **Widerrufen** â†’ tap **QR-Code** on another device (old QR) â†’ expect an invalid/expired link error

4. **Copy + QR equivalence**
   - Copy the link and scan the QR code â†’ both should resolve to the same URL
