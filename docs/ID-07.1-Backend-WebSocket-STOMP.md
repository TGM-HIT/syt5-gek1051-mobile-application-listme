## Overview

The backend uses Spring's STOMP-over-WebSocket support to enable real-time bidirectional communication between devices. All item mutation events (create, update, check, delete) are broadcast to every device currently viewing the same list, enabling sub-second collaborative sync without polling.

The implementation is split across three classes:

| Class | Role |
|-------|------|
| `WebSocketConfig` | Registers the STOMP endpoint and configures the in-memory broker |
| `SyncMessageHandler` | Receives client-initiated messages (join/leave) and routes them |
| `ListSyncBroadcaster` | Sends CRDT operation payloads and presence events to topic subscribers |

## WebSocketConfig

**File:** `config/WebSocketConfig.java`

### Endpoint

```
ws://host/ws/websocket?deviceId=<uuid>
```

SockJS is enabled as a fallback for environments where native WebSockets are blocked (corporate proxies, some mobile browsers). The SockJS endpoint is at `/ws`.

### Message broker

```java
registry.enableSimpleBroker("/topic");         // server ŌåÆ client (pub/sub)
registry.setApplicationDestinationPrefixes("/app"); // client ŌåÆ server (handler methods)
```

- **`/topic/**`** ŌĆö server pushes messages to all subscribers on this prefix. Clients subscribe to topics; they cannot publish to `/topic` directly.
- **`/app/**`** ŌĆö client-to-server messages routed to `@MessageMapping` handler methods.

### DeviceHandshakeInterceptor

The config registers `DeviceHandshakeInterceptor` as a handshake interceptor. This reads the `deviceId` query parameter from the WebSocket upgrade request and stores it in the session attributes. All subsequent message handlers retrieve the device identity from there (see [ID-07.2](ID-07.2-PresenceHandler-DeviceInterceptor.md)).

## SyncMessageHandler

**File:** `websocket/SyncMessageHandler.java`

Handles client-initiated STOMP frames:

### `@MessageMapping("/list/{listId}/join")`

Called when a device opens a list detail view. The device announces its presence so other viewers are notified.

```
Client sends STOMP SEND to: /app/list/{listId}/join
Handler reads deviceId from session attributes
Calls ListSyncBroadcaster.broadcastJoin(listId, deviceId)
  ŌåÆ updates PresenceTracker
  ŌåÆ broadcasts PresenceMessage("joined", ...) to /topic/list/{listId}/presence
```

The handler also stores the `listId` in the STOMP session attributes under key `"lastListId_{listId}"` so it can be cleaned up on disconnect.

### `@MessageMapping("/list/{listId}/leave")`

Called when a device navigates away from the list view (explicit leave). Triggers a `"left"` presence broadcast.

### `@EventListener(SessionDisconnectEvent)`

Handles abrupt disconnects (browser closed, connection dropped). Calls `PresenceTracker.disconnectDevice(deviceId)` to remove the device from all lists it was tracking. No broadcast is sent ŌĆö the topic subscriptions are already gone.

## ListSyncBroadcaster

**File:** `websocket/ListSyncBroadcaster.java`

The single outbound broadcast point. All code that needs to push to WebSocket clients uses this class.

### Topic layout

| Topic | Payload type | Purpose |
|-------|-------------|---------|
| `/topic/list/{listId}` | `CrdtOperationResponse` | Item CRDT operations (create/update/check/delete) |
| `/topic/list/{listId}/presence` | `PresenceMessage` | Join/leave/snapshot presence events |

### `broadcastOp(UUID listId, CrdtOperation op)`

Called by the HTTP item controllers after a successful DB write. Converts the saved `CrdtOperation` entity to a `CrdtOperationResponse` DTO and broadcasts it to all subscribers on `/topic/list/{listId}`.

```
POST /api/lists/{id}/items  ŌåÆ  ItemService saves item + CrdtOperation
                             ŌåÆ  ListSyncBroadcaster.broadcastOp(listId, op)
                             ŌåÆ  /topic/list/{listId} receives CrdtOperationResponse
```

### `broadcastJoin / broadcastLeave / sendSnapshot`

Presence helpers that call into `PresenceTracker` then broadcast a `PresenceMessage`. See [ID-07.2](ID-07.2-PresenceHandler-DeviceInterceptor.md) for the full presence flow.

## Configuration notes

- The in-memory broker (`enableSimpleBroker`) is suitable for single-instance deployments. A production multi-node setup would replace this with a full broker (e.g. RabbitMQ with STOMP plugin) by switching to `enableStompBrokerRelay`.
- `setAllowedOriginPatterns("*")` is intentionally permissive for development. Tighten to the production frontend origin in production config.
