# ID-03.1 — Create a new list with a name

Issue: ID 3.1
Files:
- Backend: `listme-backend/src/main/java/com/oliwier/listmebackend/api/ListController.java` (POST /api/lists)
- Frontend Store: `listme-frontend/src/stores/lists.ts` (create)
- Frontend UI: `listme-frontend/src/views/HomeView.vue` (create dialog/button)

---

## Overview

As a user, I want to create a new shopping list with a name (and optional emoji/preset). Creation works online via the REST API and offline as a local placeholder with later synchronization. Items from an optional preset are copied into the new list on the server.

---

## API design (backend)

Endpoint: POST `/api/lists`

See `ListController#create`:
- Creates a `ShoppingList` with a UUID, name from the request, and a default emoji "🛒" if none is provided
- Links the current device as `owner` via `ListDevice`
- Optional: copies items from the provided `presetId` into the new list
- Response: `ListResponse` including `itemCount` (when presets are used)
- HTTP 201 Created

Relevant implementation (conceptually):
- Persistence: `ShoppingListRepository.save(list)`
- Presets: `PresetItemRepository.findByPresetIdOrderByPosition(...)` → `ItemRepository.saveAll(items)`

---

## Frontend behavior

Store: `useListsStore.create(req)` in `src/stores/lists.ts`:
- Online:
  - `listService.create(req)`
  - Prepends the new list in the Pinia store (`unshift`)
  - Immediately writes to IndexedDB (`CacheService.saveList`)
- Offline (detected network error):
  - Generates a client-side UUID `tempId`
  - Creates a minimal local `ShoppingList` (participantCount=1, itemCount=0)
  - Saves to cache (`CacheService.saveList`) and records a pending entry in `cacheDb.pendingLists`
  - The UI shows the list immediately; sync happens later

UI: `HomeView.vue`
- Uses `useListsStore`
- Provides dialog/inputs for name (optional emoji, preset)
- Calls `await listsStore.create({ name, emoji, presetId })`

---

## Edge cases
- Empty name is rejected by request validation on the server.
- Offline: a placeholder is created; later mapping from tempId to server ID happens in the sync flow (documented separately).
- Double clicks: The store side-effects are idempotent for quick successive creates; the UI should disable the submit button while the request is in-flight.

---

## Manual verification
1. Online:
   - Open app → Home
   - Click “New list”, enter a name → Save
   - Expected: New card appears at the top; a reload still shows it (cache + API)
2. With preset:
   - Pick a preset → Create
   - Expected: `itemCount` > 0 according to preset; detail view shows copied items
3. Offline:
   - Disconnect network → Create a list
   - Expected: A list with a generated ID appears immediately; after reconnect it is synchronized with the server
