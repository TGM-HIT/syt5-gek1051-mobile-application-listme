# ID-07.4 — Reconnection Manager

**Issue:** #21
**File:** `listme-frontend/src/services/websocket.ts` (reconnect configuration)

---

## Overview

The STOMP client in `websocket.ts` uses `@stomp/stompjs`'s built-in automatic reconnection. No custom reconnection loop is written — instead, the library's `reconnectDelay` option is configured and the `ConnectionBanner` reflects the resulting state to the user.

---

## How reconnection works

When the STOMP connection drops unexpectedly (network loss, server restart, idle timeout), the `Client` instance automatically attempts to reconnect after a delay.

```typescript
client = new Client({
  brokerURL: buildBrokerUrl(deviceId),
  reconnectDelay: 1000,   // milliseconds before first retry
  onConnect:    () => console.debug('[WS] connected'),
  onDisconnect: () => console.debug('[WS] disconnected'),
  onStompError: (frame) => console.error('[WS] STOMP error', frame),
})
```

### `reconnectDelay` behaviour in `@stomp/stompjs`

The library uses `reconnectDelay` as the base interval. Internally it applies exponential backoff with jitter up to a configurable maximum:

| Attempt | Approximate wait |
|---------|-----------------|
| 1 | ~1 s |
| 2 | ~2 s |
| 3 | ~4 s |
| 4 | ~8 s |
| … | … (capped by library default) |

This prevents thundering-herd reconnections when a server restarts and many clients reconnect simultaneously.

### Reconnection lifecycle

```
Connection drops
      │
      ▼
onDisconnect fires
      │
      ▼
Client waits reconnectDelay ms
      │
      ▼
Client attempts new WebSocket upgrade
  (re-sends ?deviceId=<uuid> query param)
      │
      ├── success → onConnect fires
      │               → useListSync re-subscribes topics automatically
      │                 (STOMP subscriptions are re-established by the library
      │                  via the activationState mechanism)
      │
      └── failure → wait again (backoff) → retry
```

### Topic re-subscription

`@stomp/stompjs` does **not** automatically re-subscribe to topics after reconnect unless you configure `subscribeOnConnect` callbacks. In this app, `useListSync.startSync()` is called once on component mount and stores the subscription setup in the client's `onConnect` callback via the `Client` active subscription state — the library replays subscriptions from `beforeConnect` / `onConnect` automatically when the underlying `active` flag is set.

If the component unmounts during a reconnection attempt, `stopSync()` calls `disconnectWebSocket()` → `client.deactivate()`, which cancels the pending reconnection cleanly.

## User experience during reconnection

The `ConnectionBanner` reflects reconnection state through its three-state logic:

```
Connection drops
  → isOnline still true (network available, just server issue)
  → props.connected becomes false (useListSync sets it from isConnected())
  → Banner shows "Verbindung wird hergestellt…" (yellow, pulsing dot)

Reconnection succeeds
  → onConnect fires → connectWebSocket() promise resolves (for new callers)
  → useListSync sets connected.value = true
  → Banner shows "Verbunden" (green) for 2 seconds, then hides
```

If the network itself goes down:
```
window offline event fires
  → useOffline.isOnline = false
  → Banner shows "Kein Internet — Offline gespeichert" (red)
  → App switches to offline mode (SyncQueue + IndexedDB)
```

---

## No manual retry needed

The app does not need to manually track reconnection attempts or implement a retry loop because:

1. **`@stomp/stompjs` handles retries** — the `Client` stays active until `deactivate()` is called.
2. **Subscriptions are replayed** — the library re-sends SUBSCRIBE frames after reconnect.
3. **Offline operations are queued** — any mutations while disconnected go into `SyncQueue` (see [ID-01.3](ID-01.3-SyncQueue.md)) and flush on reconnect regardless of WebSocket state.
