# ID-18.1 — Barcode Scan

Issue: ID 18

Files to know:
- Frontend component: `listme-frontend/src/components/item/BarcodeScannerModal.vue`
- Frontend integration: `listme-frontend/src/components/item/AddItemSheet.vue`

---

## Overview

Users can add items by scanning a product barcode (EAN/UPC/QR) directly from the `AddItemSheet`. The native `BarcodeDetector` browser API handles scanning (no external library) and product details are resolved via the OpenFoodFacts API. On a successful scan the item name, quantity, unit and image URL are pre-filled in the form.

---

## Frontend

### BarcodeScannerModal.vue

A full-screen modal that opens the device camera and continuously scans for barcodes using the native `BarcodeDetector` API.

**Supported formats:** `ean_13`, `ean_8`, `upc_a`, `upc_e`, `code_128`, `code_39`, `qr_code`

**Flow:**
1. Modal opens → `getUserMedia` requests camera access (rear camera preferred via `facingMode: 'environment'`)
2. A `requestAnimationFrame` loop feeds frames into `BarcodeDetector.detect()`
3. On first detection → barcode value is sent to the **OpenFoodFacts API**:
   ```
   GET https://world.openfoodfacts.org/api/v0/product/{barcode}.json
   ```
4. Response is parsed for:
   - `product_name_de` → `product_name` → fallback to barcode value
   - `image_front_small_url` for the item image
   - `quantity` field, split into numeric quantity + unit string
5. A confirmation overlay is shown with the resolved product name and image
6. On confirm → emits `barcodeScanned` with `{ name, quantity, quantityUnit, imageUrl }`
7. On cancel → scanner resumes

**Browser support fallback:** If `window.BarcodeDetector` is undefined, the modal displays an unsupported message instead of opening the camera.

**Camera cleanup:** `MediaStream` tracks are stopped on modal close to release the camera.

### AddItemSheet.vue integration

- A barcode icon button sits in the input row of `AddItemSheet`
- Tap opens `BarcodeScannerModal` via `showScanner = true`
- `@barcodeScanned` handler populates `form.name`, `form.quantity`, `form.quantityUnit`, `form.imageUrl` and auto-focuses the name input for any manual corrections

---

## No backend changes

No backend endpoint was added or modified. Product resolution happens entirely client-side via OpenFoodFacts. The populated form fields are submitted through the normal `POST /api/lists/{listId}/items` flow (see [ID-04.5](ID-04.5-Create-Item.md)).

---

## Edge cases

- **No product found:** If the barcode exists but OpenFoodFacts has no entry, the name falls back to the raw barcode string; the user can edit it before saving
- **Network offline during scan:** The OpenFoodFacts lookup will fail; name falls back to raw barcode value, other fields remain empty
- **Ambiguous quantity string:** The `quantity` field from OpenFoodFacts (e.g. `"500 g"`) is split on the first space; irregular formats may result in the full string landing in the name field — user can correct manually
- **Permission denied:** If camera permission is denied the modal shows an error prompt

---

## Manual verification

1. **Happy path**
   - Open a list → tap the barcode button in AddItemSheet → point camera at an EAN-13 product barcode
   - Expect: product name, image and quantity pre-filled → confirm → item added to list

2. **Unknown barcode**
   - Scan a barcode not in OpenFoodFacts
   - Expect: raw barcode string used as name, no image or quantity filled

3. **Unsupported browser**
   - Open in Firefox (no BarcodeDetector support)
   - Expect: unsupported message shown instead of camera

4. **Cancel flow**
   - Open scanner → tap cancel without scanning
   - Expect: modal closes, form unchanged, camera released
