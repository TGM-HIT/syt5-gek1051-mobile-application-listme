# ID-01.4 — Service Worker & PWA Setup

**Issue:** #26
**File:** `listme-frontend/vite.config.ts` (VitePWA block)

---

## Overview

ListMe is a Progressive Web App (PWA). This means it can be installed on a device's home screen and used like a native app — including offline. The PWA functionality is powered by [vite-plugin-pwa](https://vite-pwa-org.netlify.app/), which wraps [Workbox](https://developer.chrome.com/docs/workbox/) to generate a Service Worker automatically at build time.

---

## What a Service Worker does here

A Service Worker is a background script that runs separately from the main page. In ListMe it has two jobs:

1. **Cache the app shell** — on first load, all JS, CSS, HTML, fonts, and icons are stored in the browser's Cache Storage. On subsequent loads (including offline), the Service Worker serves these files from cache instead of the network. The page loads instantly.

2. **Auto-update** — when a new version of the app is deployed, the Service Worker detects the change (`registerType: 'autoUpdate'`), installs the new version in the background, and activates it immediately via `skipWaiting: true`. Users always see the latest version without needing to manually refresh.

## Configuration

**File:** `vite.config.ts`

```typescript
VitePWA({
  registerType: 'autoUpdate',     // SW updates itself silently
  injectRegister: 'auto',         // auto-injects registration script into index.html

  devOptions: {
    enabled: true,                // SW active in dev mode too (so install prompt works)
    type: 'module',
  },

  workbox: {
    globPatterns: ['**/*.{js,css,html,ico,png,svg,woff,woff2}'],
    // ^ Precache all static assets at build time

    runtimeCaching: [{
      urlPattern: /\.(woff2|png|svg|ico)$/,
      handler: 'CacheFirst',      // serve from cache, only fetch if not found
      options: {
        cacheName: 'assets',
        expiration: {
          maxEntries: 60,
          maxAgeSeconds: 30 * 24 * 60 * 60,  // 30 days
        },
      },
    }],

    skipWaiting: true,            // new SW activates immediately (no "waiting" state)
    clientsClaim: true,           // new SW takes control of all open tabs
  },

  manifest: { ... }
})
```

### Why API responses are NOT cached by the Service Worker

The `runtimeCaching` config deliberately excludes `/api/**`. API responses are cached by `CacheService` in IndexedDB (Dexie) instead. Reasons:

- Every API request includes an `X-Device-Id` header. The HTTP Cache (used by the Service Worker) keys responses by URL only — it would serve the wrong device's data if headers differ.
- IndexedDB gives the app full control: structured queries, partial updates, TTL management.
- Separating concerns means the Service Worker handles static assets (it's good at this) and the app layer handles dynamic data (it needs to be).

## Web App Manifest

The manifest tells the browser how to present the app when installed:

```json
{
  "name": "ListMe",
  "short_name": "ListMe",
  "description": "Kollaborative Einkaufslisten — offline first",
  "theme_color": "#303446",
  "background_color": "#232634",
  "display": "standalone",
  "orientation": "portrait",
  "start_url": "/",
  "icons": [
    { "src": "/pwa-192x192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/pwa-512x512.png", "sizes": "512x512", "type": "image/png" },
    { "src": "/pwa-512x512.png", "sizes": "512x512", "type": "image/png", "purpose": "any maskable" },
    { "src": "/icon.svg",        "sizes": "any",       "type": "image/svg+xml" }
  ]
}
```

| Field | Value | Effect |
|-------|-------|--------|
| `display: standalone` | Hides the browser chrome (address bar, nav buttons) | App looks native |
| `theme_color: #303446` | Catppuccin Frappe base | Android status bar color matches app |
| `background_color: #232634` | Catppuccin Frappe crust | Splash screen background |
| `orientation: portrait` | Locks rotation | Consistent shopping list experience |
| `maskable` icon | Provided alongside standard icon | Android adaptive icons (round, squircle, etc.) |

## Install prompt

On Android Chrome and Edge, the browser shows an "Add to Home Screen" banner automatically when:
1. The app is served over HTTPS (or localhost)
2. A valid manifest is present
3. A Service Worker is registered
4. The user has visited the site at least twice

On iOS Safari, the user must manually tap **Share → Add to Home Screen**. There is no programmatic prompt on iOS.

---

## Verifying the Service Worker

In Chrome DevTools:

1. **Application → Service Workers** — shows the active SW, its state (activated / waiting), and a "Update" button
2. **Application → Cache Storage** — shows `workbox-precache-*` (static assets) and `assets` (runtime cache)
3. **Application → Manifest** — shows the parsed manifest with icon previews and installability checks
4. **Lighthouse** → run a PWA audit to see installability score

## Caching layers summary

| Layer | Technology | What it caches | Who manages it |
|-------|-----------|----------------|----------------|
| App shell | Service Worker (Workbox) | JS, CSS, HTML, fonts, icons | Auto (vite-plugin-pwa) |
| API data | IndexedDB (Dexie / CacheService) | Lists, items | App code |
| Offline ops | IndexedDB (Dexie / OperationQueue) | Pending CRDT operations | App code |
