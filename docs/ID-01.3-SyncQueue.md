## Overview

The SyncQueue is the mechanism that makes offline writes durable and eventually consistent. When the device has no internet connection, every mutation (create item, check item, rename item, etc.) is stored as a CRDT operation in IndexedDB. The moment connectivity is restored, all queued operations are sent to the server in a single batch.

---

## The problem it solves

Without the SyncQueue:

```
User adds item while offline
    ‚îÇ
    ‚ñº
POST /api/lists/{id}/items ‚Üí AxiosError: Network Error
    ‚îÇ
    ‚ñº
Item disappears. User confused.
```

With the SyncQueue:

```
User adds item while offline
    ‚îÇ
    ‚ñº
Optimistic: item added to Pinia store + IndexedDB immediately
    ‚îÇ
    ‚ñº
OperationQueue.enqueue(ITEM_CREATE op) ‚Üí persisted to IndexedDB
    ‚îÇ
    ‚ñº
(time passes, user goes about their day)
    ‚îÇ
    ‚ñº
Connectivity restored ‚Üí useSyncQueue flushes queue
    ‚îÇ
    ‚ñº
Server receives op ‚Üí item saved ‚Üí all devices see it
```

## OperationQueue

**File:** `src/crdt/OperationQueue.ts`

The `OperationQueue` is a static service backed by a dedicated Dexie database (`listme-opqueue`). It stores `CrdtOperation` objects with an additional `_synced` flag.

### Database schema

```
Database: listme-opqueue
Table:    queue
  id         string (UUID)    ‚Üê primary key
  listId     string (UUID)    ‚Üê indexed ‚Äî used to group ops by list
  createdAt  number           ‚Üê indexed ‚Äî used for chronological ordering
  _synced    0 | 1            ‚Üê indexed ‚Äî 0 = pending, 1 = synced
  + all CrdtOperation fields (operationType, payload, vectorClock, deviceId)
```

### Methods

#### `enqueue(op: CrdtOperation): Promise<void>`

Saves an operation to the queue with `_synced = 0`. Called from the Pinia stores whenever a mutation cannot reach the server.

```typescript
await OperationQueue.enqueue({
  id: crypto.randomUUID(),
  listId,
  deviceId,
  operationType: 'ITEM_CREATE',
  payload: { itemId, name: 'Milk', position: 3 },
  vectorClock: { [deviceId]: 4 },
  createdAt: Date.now(),
})
```

#### `getPending(listId: string): Promise<CrdtOperation[]>`

Returns all unsynced operations for a single list, sorted by `createdAt` ascending (oldest first = correct causal order).

#### `getAllPending(): Promise<CrdtOperation[]>`

Returns all unsynced operations across every list. Used by `useSyncQueue` during flush.

#### `markSynced(opId: string): Promise<void>`

Marks a single operation as synced (`_synced = 1`). Synced operations are kept temporarily for deduplication purposes.

#### `markAllSynced(opIds: string[]): Promise<void>`

Batch version of `markSynced`. Called after a successful flush of a list's operations.

#### `pruneOld(maxAgeMs?: number): Promise<void>`

Deletes synced operations older than `maxAgeMs` (default: 7 days). Prevents the queue from growing indefinitely. Called automatically after every successful flush.

## useSyncQueue

**File:** `src/composables/useSyncQueue.ts`

A Vue composable that watches `isOnline` (from `useOffline`) and triggers a flush whenever the device comes back online.

### Mount point

`useSyncQueue()` is called once at the app root (`App.vue`). It must not be called per-component ‚Äî it manages module-level state and the `watch` only needs to exist once.

### Flush logic

```
isOnline changes to true
    ‚îÇ
    ‚ñº
flushInProgress? ‚Üí yes ‚Üí return (prevent parallel flushes)
    ‚îÇ
    no
    ‚ñº
Set flushInProgress = true
    ‚îÇ
    ‚ñº
OperationQueue.getAllPending()
    ‚îÇ
    ‚ñº
Group ops by listId
    ‚îÇ
    ‚ñº
For each listId:
    POST /api/lists/{listId}/crdt/ops [array of ops]
        ‚îÇ
        ‚îú‚îÄ‚îÄ success ‚Üí collect opIds + listId
        ‚îî‚îÄ‚îÄ failure ‚Üí leave ops in queue, log warning, continue next list
    ‚îÇ
    ‚ñº
markAllSynced(successfulOpIds)
    ‚îÇ
    ‚ñº
pruneOld()
    ‚îÇ
    ‚ñº
For each flushed listId:
    itemsStore.fetchAll(listId) ‚Üí reconcile local state with server
    ‚îÇ
    ‚ñº
Set flushInProgress = false
```

### The `flushInProgress` guard

The module-level `flushInProgress` flag prevents concurrent flushes. Without it, rapid online/offline cycling could result in the same operations being posted multiple times. The server is idempotent on operation IDs, but avoiding redundant requests is still preferable.

### Error handling

If posting to a specific list fails (e.g. the list was deleted on the server), the error is logged and the ops for that list remain in the queue. They will be retried on the next reconnect. This is intentional ‚Äî a network error on one list should not prevent other lists from syncing.

## End-to-end flow

```
[OFFLINE]

User creates item "Bread"
    ‚îÇ
    ‚îú‚îÄ‚îÄ itemsStore adds item to Pinia with client UUID
    ‚îú‚îÄ‚îÄ CacheService.saveItem() ‚Üí IndexedDB
    ‚îú‚îÄ‚îÄ LocalClockService.getNextClock() ‚Üí { deviceId: 5 }
    ‚îî‚îÄ‚îÄ OperationQueue.enqueue({
            id: "op-uuid",
            operationType: "ITEM_CREATE",
            payload: { itemId: "item-uuid", name: "Bread", position: 2 },
            vectorClock: { "device-uuid": 5 },
            createdAt: 1709042400000
        })

[BACK ONLINE]

window 'online' event fires
    ‚îÇ
    ‚îî‚îÄ‚îÄ useSyncQueue.watch(isOnline) triggers
            ‚îÇ
            ‚îî‚îÄ‚îÄ flushQueue()
                    ‚îÇ
                    ‚îî‚îÄ‚îÄ POST /api/lists/{listId}/crdt/ops
                            [{ id: "op-uuid", operationType: "ITEM_CREATE", ... }]
                        ‚îÇ
                        ‚îî‚îÄ‚îÄ Server: SyncEngine.applyIncoming()
                                - item created in DB with client's UUID
                                - vector clock updated
                                - op broadcast via WebSocket to other devices
                        ‚îÇ
                        ‚îî‚îÄ‚îÄ markAllSynced(["op-uuid"])
                        ‚îî‚îÄ‚îÄ pruneOld()
                        ‚îî‚îÄ‚îÄ itemsStore.fetchAll(listId) ‚Üí reconcile
```

## Server-side idempotency

The server's `SyncEngine.applyIncoming()` checks `existsById(op.id)` before processing each operation. If the same operation UUID is posted twice (e.g. due to a network timeout where the first POST succeeded but the response was lost), the second post is silently ignored. This makes retries safe.
