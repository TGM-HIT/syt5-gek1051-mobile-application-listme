## Overview

Once a conflict is detected (see [ID-05.1](ID-05.1-Konflikterkennung.md)), it must be resolved so all devices converge to the same state. ListMe uses **Last-Write-Wins (LWW)** as the automatic merge strategy: the operation with the most recent timestamp is accepted; the other is discarded. A deterministic tiebreaker handles equal timestamps.

The resolved state is immediately reflected in the UI via the Pinia items store. The user is informed through a dismissible `ConflictBanner` inside the list view.

---

## LWW Resolution

**File:** `crdt/ConflictDetector.ts`

```ts
export function resolveLww(a: CrdtOperation, b: CrdtOperation): CrdtOperation
```

Resolution order:

1. **Timestamp** ‚Äî compare `payload['timestamp']` (explicit ms timestamp set by the writing device) with fallback to `op.createdAt` (server-assigned)
2. **DeviceId tiebreak** ‚Äî if timestamps are equal, the operation whose `id` is lexicographically greater wins. This is deterministic: every device performing the same comparison reaches the same conclusion without coordination.

```ts
if (tsA > tsB) return a          // A is newer ‚Üí A wins
if (tsB > tsA) return b          // B is newer ‚Üí B wins
return a.id > b.id ? a : b       // equal timestamps ‚Üí higher ID wins
```

### Why LWW is appropriate here

Shopping list operations are low-stakes mutations (rename, check, delete). The cost of occasionally discarding a concurrent update is much lower than requiring user intervention on every conflict. LWW is also idempotent and commutative when the tiebreak is deterministic, which satisfies the CRDT convergence requirement.

## What "resolved" means in practice

The items store is updated by `applyOp()` as soon as the operation arrives ‚Äî resolution is implicit in the arrival order on the WebSocket. The later-arriving operation overwrites the earlier one in the Pinia store, which matches LWW semantics.

The explicit `resolveLww()` function is available for cases where both operations need to be compared programmatically (e.g. batch replay after reconnect). The WebSocket path relies on server-side ordering, while `resolveLww()` provides a client-side fallback.

---

## Limitations

- **No manual merge UI** ‚Äî the user cannot view the two conflicting values side by side and choose one. The banner informs them that a conflict was automatically resolved but does not show what was discarded. A future iteration could add a conflict detail sheet.
- **LWW is not always semantically correct** ‚Äî if Device A checks an item and Device B simultaneously deletes it, LWW may restore the deleted item (the check arrives later). This is an accepted tradeoff for the current scope.
- **Session-scoped detection** ‚Äî `sessionOps` is reset when the user leaves the list. Conflicts that occurred while the user was not viewing the list are not surfaced retroactively.
