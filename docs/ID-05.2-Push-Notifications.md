## Overview

When someone checks off or deletes an item in a shared list, all other devices with access to that list receive a push notification — even if the app is closed, the screen is locked, or the PWA is not in the foreground.

The notification says things like:
- **"Abgehakt ✓ — Milch wurde gekauft"** (item checked)
- **"Wieder offen — Milch ist wieder offen"** (item unchecked)
- **"Artikel entfernt — Milch wurde aus der Liste entfernt"** (item deleted)

Tapping the notification opens the relevant list directly.

---

## Technology

**Web Push API** with **VAPID** (Voluntary Application Server Identification) — the standard W3C mechanism for server-initiated browser push notifications. Works in Chrome, Firefox, Edge, and Safari 16.4+, including PWAs on Android and iOS.

Key components:
- **VAPID key pair** — identifies the server to push services (e.g. FCM for Chrome, Mozilla Push Service for Firefox)
- **PushSubscription** — an endpoint + encryption keys generated by the browser, unique per device per app
- **Service Worker** — receives `push` events while the app is backgrounded or closed; shows the system notification

No FCM/APNs SDK is needed. The browser handles routing to the correct push service based on the subscription endpoint.

## Flow

```
User opens app
    │
    ▼
NotificationPermissionBanner appears (once, if permission === 'default')
    │ user taps "Aktivieren"
    ▼
Notification.requestPermission() → 'granted'
    │
    ▼
navigator.serviceWorker.ready → pushManager.subscribe(vapidPublicKey)
    │
    ▼ browser creates PushSubscription { endpoint, keys: { p256dh, auth } }
    │
    ▼
POST /api/notifications/subscribe  →  stored in push_subscriptions table
    │
    │  ... later, on another device ...
    │
    ▼
User B checks off "Milch"
    │
    ▼
ItemService.toggleCheck() → saves to DB → broadcasts CRDT op → ...
    │
    └──► NotificationService.notifyItemChecked() [async, non-blocking]
              │
              ▼
         findByListId() → filter out triggering device
              │
              ▼
         for each other device: findByDeviceId() → PushSubscription
              │
              ▼
         PushService.send(Notification) → HTTP to push endpoint (FCM / Mozilla)
              │
              ▼
         Service Worker on Device A receives `push` event
              │
              ▼
         self.registration.showNotification("Abgehakt ✓", { body: "Milch wurde gekauft", ... })
              │
              ▼
         System notification shown, even if app is closed / screen locked
```

## Backend

### VAPID Keys

VAPID is an asymmetric key pair on the P-256 elliptic curve. The **public key** is sent to the browser when it subscribes. The **private key** signs each push request so push services can verify the sender.

**Configuration** (`application.properties`):
```properties
vapid.public-key=${VAPID_PUBLIC_KEY:}
vapid.private-key=${VAPID_PRIVATE_KEY:}
vapid.subject=mailto:admin@listme.app
```

**First-run key generation:** If both properties are blank, `NotificationService.init()` auto-generates a key pair using BouncyCastle and prints them to the log at `WARN` level:

```
WARN  n.NotificationService - =======================================================
WARN  n.NotificationService -   VAPID keys not set — generating temporary keys.
WARN  n.NotificationService -   Add these to application.properties to persist them:
WARN  n.NotificationService -   vapid.public-key=BxxxxXXX...
WARN  n.NotificationService -   vapid.private-key=yyyyyYYY...
WARN  n.NotificationService - =======================================================
```

Copy the printed values into `application.properties` (or set `VAPID_PUBLIC_KEY` / `VAPID_PRIVATE_KEY` environment variables in `docker-compose.prod.yml`). **Do not change the keys after devices have subscribed** — existing subscriptions are tied to the public key and will break.

### Database

```sql
-- V11__push_subscriptions.sql
CREATE TABLE push_subscriptions (
    id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id  UUID NOT NULL REFERENCES devices(id) ON DELETE CASCADE,
    endpoint   TEXT NOT NULL,
    p256dh     TEXT NOT NULL,
    auth_key   TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (device_id, endpoint)
);
```

One row per (device, push endpoint). The `UNIQUE` constraint allows `saveSubscription` to safely upsert on repeated calls (subscription endpoints can rotate after a service worker update).

### REST API

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/notifications/vapid-public-key` | Returns the base64url-encoded VAPID public key |
| `POST` | `/api/notifications/subscribe` | Save / upsert a push subscription for the current device |
| `DELETE` | `/api/notifications/subscribe` | Remove a subscription (on unsubscribe) |

All endpoints require `X-Device-Id` header (resolved via `DeviceArgumentResolver`, same as all other endpoints).

**Subscribe request body:**
```json
{
  "endpoint": "https://fcm.googleapis.com/fcm/send/...",
  "p256dh":   "BNbwAbc123...",
  "auth":     "authSecret=="
}
```

### NotificationService

The service is `@Async` on `notifyItemChecked` and `notifyItemDeleted` — push sends happen on a thread pool after the HTTP response has already been returned. This keeps item toggle/delete latency unaffected by push latency (or push failures).

```java
@Async
public void notifyItemChecked(ShoppingList list, Item item, Device triggerDevice) {
    if (pushService == null) return;  // VAPID not configured
    // ...send to all other devices...
}
```

The triggering device is always excluded from notifications (no need to tell the user what they just did themselves).

## Frontend

### Custom Service Worker (`src/sw.ts`)

The vite config was switched from `strategies: 'generateSW'` to `strategies: 'injectManifest'` so that a hand-written service worker file can be used. Workbox still injects the precache manifest via `self.__WB_MANIFEST`.

```typescript
// Handle push events — fires even when the app is closed / screen is locked
self.addEventListener('push', (event) => {
  const data = event.data.json()
  event.waitUntil(
    self.registration.showNotification(data.title, {
      body: data.body,
      icon: '/pwa-192x192.png',
      tag: `listme-${data.listId}`,  // groups notifications per list
      renotify: true,
      vibrate: [200, 100, 200],
    })
  )
})

// Handle notification tap — focus existing tab or open new one
self.addEventListener('notificationclick', (event) => {
  event.notification.close()
  const url = event.notification.data?.url ?? '/'
  event.waitUntil(
    self.clients.matchAll({ type: 'window' }).then((clients) => {
      for (const client of clients) {
        if ('focus' in client) return client.focus()
      }
      return self.clients.openWindow(url)
    })
  )
})
```

The `tag: \`listme-${data.listId}\`` groups all notifications for the same list into one slot. A rapid sequence of checkoffs won't flood the notification tray.

### pushNotification.ts

Handles the full subscription lifecycle:

```typescript
export async function subscribeToPush(): Promise<void> {
  // 1. Guard: browser support + permission
  if (!('serviceWorker' in navigator) || !('PushManager' in window)) return
  const permission = await Notification.requestPermission()
  if (permission !== 'granted') return

  // 2. Re-use existing subscription if already subscribed
  const registration = await navigator.serviceWorker.ready
  const existing = await registration.pushManager.getSubscription()
  if (existing) { await syncWithBackend(existing); return }

  // 3. Fetch VAPID public key → subscribe → sync with backend
  const { data: publicKey } = await apiClient.get('/notifications/vapid-public-key')
  const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true,
    applicationServerKey: urlBase64ToUint8Array(publicKey),
  })
  await syncWithBackend(subscription)
}
```

`subscribeToPush` is also called on every app load (in `App.vue`) when permission is already `'granted'`. This re-syncs the subscription endpoint with the backend, since push endpoints can rotate after a service worker update.

### NotificationPermissionBanner

A dismissible banner shown **once** when `Notification.permission === 'default'` (not yet decided). It never auto-requests permission — the browser requires user interaction first.

```
┌────────────────────────────────────────────────────────────────┐
│ 🔔  Benachrichtigungen aktivieren, um informiert zu werden ...  │  [Aktivieren]  [✕]
└────────────────────────────────────────────────────────────────┘
```

Tapping **Aktivieren** calls `Notification.requestPermission()` and then `subscribeToPush()`. Tapping ✕ dismisses permanently for the session.

## Notification triggers

| Action | Title | Body |
|--------|-------|------|
| Item checked | `Abgehakt ✓` | `<name> wurde gekauft` |
| Item unchecked | `Wieder offen` | `<name> ist wieder offen` |
| Item deleted | `Artikel entfernt` | `<name> wurde aus der Liste entfernt` |

Triggered in `ItemService.toggleCheck()` and `ItemService.delete()`. The `create` and `update` actions do not trigger notifications (adding to your own list is expected; renaming an item is low-priority).

## wsConnected bug fix (related)

As part of this issue, a bug was fixed where `ConnectionBanner` remained stuck on "Verbindung wird hergestellt…" even after the WebSocket had connected.

**Root cause:** `useListSync` captured the connection state once with `connected.value = isConnected()` right after `connectWebSocket()` resolved. This snapshot was never updated — so any reconnect after a drop left `connected` permanently `false`.

**Fix:** A reactive `wsConnected` ref was added to `websocket.ts` and updated directly inside `onConnect` / `onDisconnect`. `useListSync` now returns this shared ref instead of a local snapshot:

```typescript
// websocket.ts
export const wsConnected = ref(false)

client.onConnect = () => {
  wsConnected.value = true   // ← always up-to-date, no snapshots
  // ...
}
client.onDisconnect = () => {
  wsConnected.value = false
  scheduleReconnect()
}

// useListSync.ts — was: connected.value = isConnected()
return { connected: wsConnected, ... }  // live ref, not snapshot
```

This ensures the banner correctly reflects every reconnect cycle without any extra wiring.
