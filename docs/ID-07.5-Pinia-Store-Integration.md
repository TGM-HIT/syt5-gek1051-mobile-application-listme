# ID-07.5 — Frontend Pinia Store Integration

**Issue:** #22
**Files:** `listme-frontend/src/composables/useListSync.ts`, `listme-frontend/src/stores/presence.ts`, `listme-frontend/src/stores/items.ts`

---

## Overview

`useListSync` is the composable that wires together the WebSocket service, the CRDT conflict detector, and the Pinia stores. It is the single integration point a view needs to call to get real-time sync for a list.

```typescript
// ListDetailView.vue
const { connected, conflicts, dismissConflicts, startSync } = useListSync()
onMounted(() => startSync(listId))
```

When mounted, `useListSync` connects to the WebSocket, subscribes to the list's CRDT operation and presence topics, and patches the Pinia stores directly as messages arrive — no HTTP refetch required.

---

## useListSync composable

**File:** `src/composables/useListSync.ts`

### Returned state

| Ref | Type | Description |
|-----|------|-------------|
| `connected` | `Ref<boolean>` | `true` after the STOMP connection is established |
| `conflicts` | `Ref<Conflict[]>` | Conflicts detected in the current session's operation stream |

### Returned functions

| Function | Description |
|----------|-------------|
| `startSync(listId)` | Connects WebSocket, subscribes to topics, announces presence |
| `stopSync()` | Sends leave, unsubscribes all topics, clears session state |
| `dismissConflicts()` | Clears the `conflicts` ref (user has acknowledged) |

`stopSync` is also called automatically via `onUnmounted` — no manual cleanup needed from the view.

## startSync flow

```
startSync(listId)
      │
      ▼
connectWebSocket()           ← idempotent; reuses existing connection
      │
      ├── fails (offline) → log warning, return early
      │                      (app continues in offline mode)
      │
      └── succeeds
            │
            ▼
      subscribe /topic/list/{listId}          ← CRDT op stream
      subscribe /topic/list/{listId}/presence ← presence events
            │
            ▼
      send /app/list/{listId}/join            ← announce presence
```

### Session operation log

Each incoming CRDT operation is pushed to a local `sessionOps` array. After every push, `detectConflicts(sessionOps)` is called to check for concurrent vector clocks across the session. If any are found, the `conflicts` ref is updated, which triggers the `ConflictBanner` in `ListDetailView`.

---

## Incoming CRDT operation handling

```typescript
subscribe(`/topic/list/${listId}`, (payload) => {
  const op = payload as CrdtOperation
  if (op.deviceId === myDeviceId) return  // skip own ops (already applied via HTTP)
  sessionOps.push(op)
  conflicts.value = detectConflicts(sessionOps)
  applyOp(listId, op, itemsStore)
})
```

Own-device operations are filtered out because the HTTP response from the REST API already applied them to the store optimistically. Applying them again from WebSocket would cause a double-update.

### applyOp — operation types

The `applyOp` function patches the `itemsByList[listId]` array in the items store directly, without triggering an API refetch:

| `operationType` | Payload fields | Action |
|----------------|---------------|--------|
| `ITEM_CREATE` | `itemId`, `name`, `position` | Appends a new `Item` if not already present (idempotent) |
| `ITEM_CHECK` | `itemId`, `checked` | Patches `checked` on the matching item in-place |
| `ITEM_UPDATE` | `itemId`, `name` | Patches `name` on the matching item in-place |
| `ITEM_DELETE` | `itemId` | Filters the item out of the array |

For `ITEM_CREATE`, a full `Item` object is constructed from the op payload with sensible defaults for fields not carried in the op (e.g. `labels: []`, `quantity: null`). This keeps the store's item shape consistent with what the HTTP API returns.

## Incoming presence event handling

```typescript
subscribe(`/topic/list/${listId}/presence`, (payload) => {
  if (msg.event === 'snapshot') presenceStore.setSnapshot(listId, msg.onlineDevices)
  else if (msg.event === 'joined') presenceStore.addDevice(listId, msg.deviceId)
  else if (msg.event === 'left')   presenceStore.removeDevice(listId, msg.deviceId)
})
```

The presence store (`src/stores/presence.ts`) keeps a `Map<listId, Set<deviceId>>`. `ListDetailView` reads `presenceStore.getCount(listId)` to show the live viewer count.

## stopSync flow

```
stopSync()
      │
      ├── send /app/list/{listId}/leave   ← notify server/other devices
      │
      ├── call each unsubscribe function  ← clean up STOMP subscriptions
      │
      ├── reset sessionOps = []
      │
      └── reset conflicts.value = []
```

The WebSocket connection itself is **not** closed on `stopSync` — only subscriptions are removed. The connection stays alive in case the user navigates to another list (the next `startSync` call reuses it). `disconnectWebSocket()` is only called explicitly if needed (e.g. app unmount).

## Data flow summary

```
Other device mutates an item
        │
        ▼ (HTTP → backend → DB → ListSyncBroadcaster)
/topic/list/{listId}  →  CrdtOperation message
        │
        ▼
useListSync: filter own ops → push to sessionOps → detectConflicts
        │
        ▼
applyOp() patches itemsStore.itemsByList[listId] in-place
        │
        ▼
Vue reactivity: ListDetailView re-renders updated item row
```

No component needs to call `fetchItems()` — the store is kept live by the WebSocket stream.
