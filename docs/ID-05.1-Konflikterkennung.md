## Overview

Conflict detection is built on **vector clocks** — logical counters that track causality across devices without requiring synchronised wall clocks. Every CRDT operation carries the full vector clock snapshot of the device that produced it. Two operations on the same item are a conflict if and only if their clocks are **CONCURRENT**: neither happened before the other.

The same logic is implemented identically in both the backend (Java) and the frontend (TypeScript) so that either side can detect conflicts independently.

---

## Vector Clock

**Backend:** `crdt/VectorClock.java`  
**Frontend:** `crdt/VectorClock.ts`

A vector clock is a `Map<deviceId, counter>`. Each device increments its own counter on every mutation; counters for other devices are never changed locally.

### Clock operations

| Method | Description |
|--------|-------------|
| `increment(deviceId)` | Returns a new clock with this device's counter increased by 1 |
| `merge(other)` | Returns a new clock with the max counter per device (used on sync) |
| `compare(other)` | Returns the causal relationship (see below) |

### Causal relations

`compare(other)` compares `this` against `other` by iterating over all device keys from both clocks:

| Relation | Meaning | Condition |
|----------|---------|-----------|
| `BEFORE` | `this` happened before `other` | All counters in `this` ≤ `other`, at least one strictly < |
| `AFTER` | `this` happened after `other` | All counters in `this` ≥ `other`, at least one strictly > |
| `EQUAL` | Identical clocks | All counters equal |
| `CONCURRENT` | Neither dominates | `this` is greater on at least one device AND `other` is greater on at least one different device |

Only `CONCURRENT` indicates a conflict.

### Example

```
Device A edits item X offline:  clockA = { A: 3, B: 1 }
Device B edits item X offline:  clockB = { A: 2, B: 2 }

compare clockA vs clockB:
  A: 3 > 2  →  thisHasGreater = true
  B: 1 < 2  →  otherHasGreater = true
  Result: CONCURRENT → conflict
```

## Backend: ConflictDetector

**File:** `crdt/ConflictDetector.java`

A stateless utility class. Takes a flat list of `CrdtOperation` entities and returns every pair that targets the same item with a `CONCURRENT` clock relationship.

```java
public static List<Conflict> detect(List<CrdtOperation> ops)
```

**Algorithm:** O(n²) pairwise scan. For each pair (i, j):
1. Check `sameTarget()` — both operations must reference the same `itemId` in their payload
2. Reconstruct `VectorClock` objects from the stored JSONB maps
3. Call `clockA.compare(clockB)` — add to result list only if `CONCURRENT`

The `Conflict` record holds both operations:
```java
public record Conflict(CrdtOperation a, CrdtOperation b) {}
```

### Integration with SyncEngine

`SyncEngine.record()` persists every mutation with a vector clock snapshot. When `getOperationsSince()` returns a list of operations to a client, callers can pass that list to `ConflictDetector.detect()` to identify which operations arrived concurrently.

The sync engine's `SyncResult` record bundles both the applied operations and any detected conflicts:

```java
public record SyncResult(List<CrdtOperation> applied, List<ConflictDetector.Conflict> conflicts) {}
```

## Frontend: ConflictDetector

**File:** `crdt/ConflictDetector.ts`

Mirrors the backend logic exactly. The frontend runs the same pairwise O(n²) scan over the `sessionOps` buffer maintained in `useListSync`.

```ts
export function detectConflicts(ops: CrdtOperation[]): Conflict[]
```

`sessionOps` accumulates every incoming CRDT operation from other devices during the current list view session. After each new operation arrives, `detectConflicts(sessionOps)` is called again — this means the conflict count can only grow within a session (it is reset when the list is left and `stopSync()` is called).

### Wiring in useListSync

```
WebSocket /topic/list/{id}  →  incoming CrdtOperation
                                    │
                                    ▼
                           sessionOps.push(op)
                                    │
                                    ▼
                     detectConflicts(sessionOps)
                                    │
                         ┌──────────┴──────────┐
                    no new conflicts       new conflicts
                         │                     │
                    conflicts.value      conflicts.value updated
                    unchanged            + notificationsStore.add()
```

Own-device operations are filtered out before pushing to `sessionOps` (compared by `deviceId`), so a device never detects a conflict with itself.

## What triggers a conflict (concrete scenario)

1. Device A and Device B both open the same list while online
2. The server goes offline (or both devices lose connectivity)
3. Device A renames item "Milch" → "Vollmilch" and saves locally
4. Device B changes the same item to "checked: true" while offline
5. Both devices reconnect and push their operations
6. The server receives both operations with `CONCURRENT` vector clocks on the same `itemId`
7. `ConflictDetector.detect()` returns one `Conflict(opA, opB)`
8. The conflict is resolved by the merge strategy (see [ID-05.3](ID-05.3-Konflikt-Resolution.md))
