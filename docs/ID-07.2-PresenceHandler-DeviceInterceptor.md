## Overview

Presence tracking lets every device know how many (and which) other devices are currently viewing the same shopping list. It powers the live participant count shown in `ListDetailView`.

Two components work together:

| Component | Responsibility |
|-----------|---------------|
| `DeviceHandshakeInterceptor` | Reads `deviceId` from the WebSocket upgrade URL and attaches it to the STOMP session |
| `PresenceTracker` | In-memory map of `listId → Set<deviceId>` — the source of truth for who is online |
| `PresenceMessage` | Record (DTO) sent over WebSocket to inform clients of presence changes |

`ListSyncBroadcaster` (documented in [ID-07.1](ID-07.1-Backend-WebSocket-STOMP.md)) uses `PresenceTracker` as its data source and `PresenceMessage` as its payload.

## DeviceHandshakeInterceptor

**File:** `websocket/DeviceHandshakeInterceptor.java`

### Purpose

The STOMP protocol itself carries no concept of device identity. This interceptor bridges the gap by reading the `deviceId` from the WebSocket upgrade request's query string and storing it in the WebSocket session attributes.

### How it works

```
Browser sends:  GET /ws/websocket?deviceId=<uuid>  (WebSocket upgrade)
                │
                ▼
DeviceHandshakeInterceptor.beforeHandshake()
  → reads ?deviceId from URI
  → attributes.put("deviceId", "<uuid>")
                │
                ▼
STOMP session established; attributes available to all @MessageMapping handlers
via SimpMessageHeaderAccessor.getSessionAttributes().get("deviceId")
```

The interceptor always returns `true` (allows the handshake). A missing `deviceId` is permitted — it just means the connection won't be tracked for presence. This avoids breaking the WebSocket connection for clients that don't send the param.

### Attribute key

```java
public static final String DEVICE_ID_ATTR = "deviceId";
```

All message handlers reference this constant rather than the raw string `"deviceId"`.

## PresenceTracker

**File:** `websocket/PresenceTracker.java`

### Data structure

```java
Map<UUID, Set<String>> listToDevices  // listId → Set of deviceId strings
```

Both the outer map and the inner sets use `ConcurrentHashMap` / `ConcurrentHashMap.newKeySet()` so reads and writes from different STOMP session threads are safe without explicit locking.

### API

| Method | Description |
|--------|-------------|
| `join(listId, deviceId)` | Adds `deviceId` to the set for `listId`. Creates the set if absent. |
| `leave(listId, deviceId)` | Removes `deviceId`. If the set becomes empty, removes the `listId` entry entirely to avoid memory leaks. |
| `disconnectDevice(deviceId)` | Iterates all lists and removes the device. Called on abrupt disconnect. Removes empty list entries. |
| `getOnlineDevices(listId)` | Returns an unmodifiable view of the current set. |

### Memory lifecycle

Lists are evicted from the map automatically when their last device leaves. This means a list with no active viewers uses zero memory in the presence tracker.

## PresenceMessage

**File:** `websocket/PresenceMessage.java`

A Java record — the JSON payload sent to `/topic/list/{listId}/presence`.

```java
record PresenceMessage(
    String event,           // "joined" | "left" | "snapshot"
    String deviceId,        // the device that triggered the event
    Set<String> onlineDevices  // full set of currently online devices for this list
)
```

The `onlineDevices` set is always included in every message so clients can reconcile their local state without needing to track incremental deltas. This makes the presence channel eventually consistent even if a message is missed.

## Presence event flow

### Device opens a list

```
useListSync.startSync(listId)
  → STOMP SEND to /app/list/{listId}/join
        │
        ▼
SyncMessageHandler.join()
  → ListSyncBroadcaster.broadcastJoin(listId, deviceId)
        │
        ├── PresenceTracker.join(listId, deviceId)
        │
        └── messaging.convertAndSend(
              "/topic/list/{listId}/presence",
              PresenceMessage("joined", deviceId, [deviceA, deviceB, ...])
            )
        │
        ▼
All subscribers receive the message; frontend adds deviceId to its presence set
```

### Device navigates away (explicit)

```
useListSync.stopSync()
  → STOMP SEND to /app/list/{listId}/leave
  → SyncMessageHandler.leave()
  → ListSyncBroadcaster.broadcastLeave()
  → PresenceTracker.leave()
  → PresenceMessage("left", ...)  broadcast
```

### Connection dropped (abrupt)

```
SessionDisconnectEvent fires
  → SyncMessageHandler.onDisconnect()
  → PresenceTracker.disconnectDevice(deviceId)
  (no broadcast — subscriptions are already torn down)
```

## Frontend consumption

The presence messages are consumed by `useListSync` and routed to the `presence` Pinia store:

```typescript
subscribe(`/topic/list/${listId}/presence`, (payload) => {
  if (msg.event === 'snapshot') presenceStore.setSnapshot(listId, msg.onlineDevices)
  else if (msg.event === 'joined') presenceStore.addDevice(listId, msg.deviceId)
  else if (msg.event === 'left')   presenceStore.removeDevice(listId, msg.deviceId)
})
```

`ListDetailView` reads `presenceStore.getCount(listId)` to display the live participant count. See [ID-07.5](ID-07.5-Pinia-Store-Integration.md) for the full frontend integration.
