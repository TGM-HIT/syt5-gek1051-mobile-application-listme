# ID-01.1 — IndexedDB Schema with Dexie

**Issue:** #23
**Files:** `listme-frontend/src/services/db.ts`, `listme-frontend/src/services/clock.ts`

---

## Overview

All persistent client-side data lives in two IndexedDB databases managed by [Dexie.js](https://dexie.org). Dexie is a typed wrapper around the browser's native IndexedDB API that provides Promises, versioned schema migrations, and compound key support.

ListMe uses two separate databases to separate concerns:

| Database | Dexie class | Purpose |
|----------|-------------|---------|
| `listme-cache` | `ListMeCacheDb` | Shopping lists and items (mirror of server state) |
| `listme-opqueue` | `OperationQueueDb` | Pending CRDT operations to sync when back online |

The operation queue database is documented in [ID-01.3-SyncQueue.md](ID-01.3-SyncQueue.md).

## listme-cache database

**File:** `src/services/db.ts`

### Tables

#### `lists`

Stores a local copy of every `ShoppingList` the device has access to.

| Column | Type | Index | Description |
|--------|------|-------|-------------|
| `id` | `string` (UUID) | Primary key | List UUID from server |
| `_savedAt` | `number` | Index | Unix timestamp (ms) of when this row was written. Used to sort most-recently-updated first. |
| *(all ShoppingList fields)* | — | — | `name`, `emoji`, `shareToken`, `itemCount`, etc. |

The `_savedAt` field is added by `CacheService` at write time and stripped before the data is returned to callers — it is a cache-internal field, invisible to the rest of the app.

#### `items`

Stores a local copy of all items across all lists.

| Column | Type | Index | Description |
|--------|------|-------|-------------|
| `id` | `string` (UUID) | Primary key | Item UUID from server |
| `listId` | `string` (UUID) | Index | Foreign key — used for fast per-list queries |
| `_savedAt` | `number` | Index | Write timestamp (same pattern as `lists`) |
| *(all Item fields)* | — | — | `name`, `checked`, `position`, `quantity`, etc. |

The `listId` index is critical for performance — `CacheService.getItems(listId)` uses a `where('listId').equals(listId)` query which hits the index directly.

#### `localClocks`

Stores the current vector clock counter per `(listId, deviceId)` pair. Used to attach vector clocks to operations that are queued while offline.

| Column | Type | Index | Description |
|--------|------|-------|-------------|
| `[listId+deviceId]` | compound | Primary key | Dexie compound key — uniquely identifies a clock per list per device |
| `listId` | `string` | (part of PK) | Which list this clock belongs to |
| `deviceId` | `string` | (part of PK) | Which device owns this counter |
| `counter` | `number` | — | Monotonically increasing integer, incremented on every offline write |

### Schema versioning

Dexie uses explicit version numbers to handle schema migrations. If a user's browser has an older version of the database, Dexie migrates it automatically on next open.

```
Version 1 → lists, items (original schema)
Version 2 → + localClocks table added
```

When adding a new table or index in the future, increment the version and add a new `.version(N).stores({...})` block — **never modify an existing version** as it would break existing user databases.

## LocalClockService

**File:** `src/services/clock.ts`

### Purpose

Provides monotonically increasing vector clock counters for CRDT operations created while offline. Each time the device creates an operation without connectivity, it needs to attach a vector clock so the server can detect causality.

### How it works

```
getNextClock(listId, deviceId)
    │
    ▼
Read current row from localClocks table
    │
  not found ──► start at counter = 0
    │
    ▼
counter++
    │
    ▼
Write updated row back to localClocks
    │
    ▼
Return { [deviceId]: counter }
```

The returned object is the format expected by `CrdtOperation.vectorClock` — a map of `{ deviceId: counter }`.

### Example

```typescript
import { LocalClockService } from '../services/clock'
import { getDeviceId } from '../services/device'

const deviceId = await getDeviceId()
const clock = await LocalClockService.getNextClock(listId, deviceId)
// clock = { "a1b2c3d4-...": 7 }
```

## Why two separate Dexie instances?

The cache (`listme-cache`) and the operation queue (`listme-opqueue`) are in separate databases so they can be opened, cleared, or inspected independently.

- Clearing the cache (e.g. on sign-out or debug reset) does not lose unsynced operations.
- The operation queue can be wiped after a full re-sync without touching the item cache.
- Separate databases avoid transaction contention between read-heavy cache reads and write-heavy queue operations.

## Inspecting the database in DevTools

1. Open Chrome DevTools → **Application** tab
2. Expand **IndexedDB** in the left sidebar
3. You'll see `listme-cache` and `listme-opqueue` as separate entries
4. Click any table to browse rows, check `_savedAt` timestamps, or manually delete rows for debugging
