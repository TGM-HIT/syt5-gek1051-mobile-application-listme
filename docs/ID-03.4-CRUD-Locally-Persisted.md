## Overview

All CRUD operations for lists (and items) are persisted locally so the app is offline-first, starts quickly, and remains usable during network issues. The persistence layer is based on IndexedDB via Dexie and follows a write-through strategy: after a successful API request, the local cache is always updated as well. When loading, data is shown from the cache first and then replaced by the server response.

---

## Architecture building blocks

1) CacheService (`src/services/cache.ts`)
- Methods: `saveLists`, `saveList`, `getLists`, `removeList`, `saveItems`, `saveItem`, `getItems`, `removeItem`, `patchItem`
- Stores snapshots in the Dexie database `listme-cache`
- Adds an internal `_savedAt` timestamp (for "last synced" sorting)

2) Stores (`src/stores/*.ts`)
- `lists.ts`:
  - `fetchAll()` first loads `CacheService.getLists()` â†’ instant UI rendering
  - After a successful `listService.getAll()` â†’ `CacheService.saveLists(fresh)`
  - `create`, `update`, `remove`, `duplicate` each write to the cache
  - Offline create: creates a temporary list with `crypto.randomUUID()` and additionally enqueues an entry in `cacheDb.pendingLists`
- `items.ts`:
  - Same pattern for items (cache-first, then API; individual patching via `patchItem`)

3) Operation Queue (`listme-opqueue`)
- For offline operations that will be synchronized with the server, `useSyncQueue` manages pending CRDT operations (see ID-01.3)

4) PWA/Service Worker
- Static caching (app shell) via Workbox, configured in `vite.config.ts`
- API responses are deliberately not cached by the SW (conflicts with X-Device-Id headers); instead, data caching happens exclusively via IndexedDB

## Lifecycles per operation

- Create (list):
  - Online: `listService.create` â†’ `lists.unshift` â†’ `CacheService.saveList`
  - Offline: create tempId list locally â†’ `CacheService.saveList` â†’ `cacheDb.pendingLists.put(...)` â†’ later sync replaces tempId with server ID

- Read (lists/items):
  - Render immediately from `CacheService.get*()`
  - Then load via API and persist via `CacheService.save*()`

- Update (rename/emoji):
  - `listService.update` â†’ replace in store â†’ `CacheService.saveList`

- Delete (list):
  - `listService.delete` â†’ remove from store â†’ `CacheService.removeList` (also deletes all items of the list)

## Consistency & sorting
- Lists returned by `getLists()` are ordered by `_savedAt` descending â†’ most recently synced on top
- Items returned by `getItems(listId)` are sorted by `position`
- Individual field changes (e.g., `checked`) are efficiently applied via `patchItem`
