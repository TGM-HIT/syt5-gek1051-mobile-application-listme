## Overview

When multiple devices have a list open at the same time, checkbox changes made on one device must appear on all others without a manual reload. This is implemented via a STOMP-over-WebSocket subscription. When another device toggles an item, the server broadcasts a `CRDT_OPERATION` message with `operationType: 'ITEM_CHECK'` on the list's topic. The `useListSync` composable receives the message and patches the Pinia store in-place, producing an immediate reactive UI update.

## WebSocket Transport

**File:** `src/services/websocket.ts`

The app connects to a STOMP broker over a standard WebSocket. The connection URL embeds the device ID so the server can filter self-echo messages.

```typescript
function buildBrokerUrl(deviceId: string): string {
  const protocol = location.protocol === 'https:' ? 'wss' : 'ws'
  return `${protocol}://${location.host}/ws/websocket?deviceId=${deviceId}`
}
```

| Function | Purpose |
|----------|---------|
| `connectWebSocket()` | Opens STOMP connection; idempotent |
| `subscribe(topic, cb)` | Returns an unsubscribe function |
| `send(destination, body)` | Publishes a message to a STOMP destination |
| `disconnectWebSocket()` | Closes the connection |

The client reconnects automatically with a 1-second delay (`reconnectDelay: 1000`).

## Composable: `useListSync`

**File:** `src/composables/useListSync.ts`

`useListSync` is called from `ListDetailView` on mount. It manages the WebSocket lifecycle, subscribes to the list's CRDT operation topic, and dispatches incoming operations to the store.

### Lifecycle

```
ListDetailView.onMounted
      │
      ▼
startSync(listId)
      │
      ├── connectWebSocket()
      │
      ├── subscribe(`/topic/list/${listId}`, handleOp)
      │
      ├── subscribe(`/topic/list/${listId}/presence`, handlePresence)
      │
      └── send(`/app/list/${listId}/join`)   ← announces presence

ListDetailView.onUnmounted
      │
      ▼
stopSync()
      │
      ├── send(`/app/list/${listId}/leave`)
      └── unsubscribe all
```

### Handling Incoming `ITEM_CHECK` Operations

Ops from the current device are silently ignored (they were already applied via the HTTP response):

```typescript
const unsubOps = subscribe(`/topic/list/${listId}`, (payload) => {
  const op = payload as CrdtOperation
  if (op.deviceId === myDeviceId) return   // ← skip own ops
  sessionOps.push(op)
  conflicts.value = detectConflicts(sessionOps)
  applyOp(listId, op, itemsStore)
})
```

### `applyOp` — ITEM_CHECK case

```typescript
case 'ITEM_CHECK': {
  const itemId = payload['itemId'] as string
  const checked = payload['checked'] as boolean
  const idx = items.findIndex(i => i.id === itemId)
  if (idx !== -1) items[idx] = { ...items[idx]!, checked }
  break
}
```

The direct array mutation triggers Vue's reactivity, updating `ItemRow` styling immediately without any store action or cache write — the server is already the source of truth for the new state.

## Conflict Detection

**File:** `src/crdt/ConflictDetector.ts`

All incoming CRDT ops for the session are accumulated in `sessionOps`. After each new op, `detectConflicts` scans for concurrent writes to the same item using vector clock comparison. If two devices toggle the same item without having seen each other's clock, a conflict is flagged and shown via `ConflictBanner`.

```typescript
conflicts.value = detectConflicts(sessionOps)
```

Conflicts are informational only — the last-write-wins value from the server is already applied. The user can dismiss the banner via `dismissConflicts()`.

## Connection State & UI

`useListSync` exposes a `connected` ref that `ListDetailView` passes to `ConnectionBanner`:

```html
<ConnectionBanner :connected="syncConnected" />
```

The banner renders a subtle indicator when the WebSocket is disconnected, so the user knows real-time updates are paused. Toggling still works offline via the CRDT queue (see ID-02.2).

## Message Flow Diagram

```
Device A taps checkbox
        │
        ▼
PATCH /lists/:listId/items/:itemId/check
        │
        ▼ (server applies change, broadcasts)
STOMP: /topic/list/:listId
  { operationType: 'ITEM_CHECK', itemId, checked, deviceId: A }
        │
        ├──► Device A  →  ignored (own deviceId)
        │
        └──► Device B  →  applyOp → items[idx].checked = true
                           → Vue reactivity → ItemRow re-renders
```

## Acceptance Criteria — Mapping

| Acceptance Criterion | Implementation |
|----------------------|----------------|
| Checkbox-Änderung erscheint auf anderen Geräten sofort | `applyOp('ITEM_CHECK')` patches Pinia store from STOMP message |
| Eigene Änderungen nicht doppelt angewendet | `if (op.deviceId === myDeviceId) return` guard in subscriber |
| Verbindungsstatus sichtbar | `ConnectionBanner` binds to `useListSync().connected` |
| Konflikte erkennbar | `ConflictBanner` shows when `detectConflicts(sessionOps)` returns results |
| Cleanup bei Verlassen der View | `onUnmounted(stopSync)` unsubscribes all STOMP topics |
