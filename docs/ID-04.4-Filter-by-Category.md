# ID-04.4 — Filter by category (NOT IMPLEMENTED)

Issue: ID 4.4

Files to know:
- Types: `listme-frontend/src/types/index.ts` (Item carries category fields)
- Store: `listme-frontend/src/stores/items.ts` (fetch + local cache)
- UI: `listme-frontend/src/views/ListDetailView.vue` (location for search and future filters)
- Backend: `ItemController.getItems` (could accept a `categoryId` param in the future)

---

## Current status

- Category-based filtering in the UI is NOT implemented as of this version.
- Users can search text across items (see ID-04.3), and items may display a category chip, but there is no UI control to filter items by a selected category.

---

## Proposed implementation (future)

UI/UX:
- Add a horizontal chip row above the items list showing available categories for the current list ("All", then each category).
- Tapping a chip sets the active filter; active chip is highlighted.
- Combine with the existing search: results are the intersection of text search and selected category.

State management:
- Pinia `items` store (or view-local state) keeps `activeCategoryId: string | null`.
- `filteredItems` computed derives from `items` by applying both `searchQuery` and `activeCategoryId`.

Backend:
- Optional: extend `GET /api/lists/{listId}/items` to accept `categoryId` to reduce payloads for large lists.
- Keep `q` + `categoryId` combinable.

Caching & offline:
- No change: items continue to live in IndexedDB via `CacheService`. Filtering is client-side for instant feedback and offline support.

---

## Manual verification (once implemented)

1) Tap a category chip (e.g., "Dairy")
- Expect the list to only show items in that category.

2) Combine with search
- Enter text and keep a category selected. Expect results to meet both criteria.

3) Clear filter
- Tap "All" or the active chip again. Expect to see all items.
